# Fire Drill: Releases

**Date:** 2025-04-22

## Scenario

Practice deploying an older version (rollback) of the Batch & Matrix service when a release introduces issues and must be reverted.

## Steps Performed

### Dictionary

- **RELEASE CANDIDATE** — the INFRA BRANCH that is ready for release.
  - Such an infra branch uses specific Docker images (not the latest).
  - Such an infra branch has been tested and is passing all tests (unit, acceptance, compatibility).

### Rollback Procedure

1. **Determine the current release number** using one of:
   - **Azure Portal:** Find the newest `aks<number>` deployment.
   - **[Grafana](https://grafana.com/docs/grafana/latest/):** Identify which executor base name is processing data. Use the `basename` selector.
     - [Executor dashboard](https://grafana.prod.batch.tt4.nl/d/batch2-executor-batch12-1/executor-batch1-2?orgId=1)

2. **Check `RELEASES.md`** in the repository:
   - Determine the previous release numbers.
   - Find the `INFRA_BRANCH` for the previous (target) deployment.

3. **Trigger Jenkins rollback job:**
   - Navigate to: Jenkins > Batch > Prod > Batch_Release > Simple_Batch_release
   - URL: [Simple\_Batch\_release job](https://ci.dev.batch.tt4.nl/view/Batch/view/Prod/view/Batch_Release/view/Simple_Batch_release/)
   - Enter the target `INFRA_BRANCH` value into the `RELEASE_CANDIDATE_BRANCH` field.
   - Enable `PANIC_MODE` to speed up the deployment when time is critical.
   - Wait for the job to complete — approximately 30 minutes.

## Lessons Learned

- On-call engineers must be able to deploy an older version when a release must be rolled back.
- The `RELEASES.md` file is the key artifact for identifying which infra branch corresponds to a given release.
- Use `PANIC_MODE` in Jenkins to accelerate deployment when speed is critical.
