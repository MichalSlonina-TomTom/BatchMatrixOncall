## Overview

The **jumphost** is a bastion VM that gates access to the Batch & Matrix production infrastructure — [AKS](https://learn.microsoft.com/en-us/azure/aks/intro-kubernetes) clusters, Azure resources, and internal tooling. All production access must go through the jumphost; there is no direct path.

Two jumphost VMs are available:

| Domain | Location | Purpose |
|--------|----------|---------|
| `jumphost.prod.batch.tt4.nl` | West Europe | Primary VM |
| `jumphost2.prod.batch.tt4.nl` | East US | Backup VM |

**Why PIM is needed:** Production subscriptions follow TomTom CCoE's OCD-compliant model. Elevated [Azure RBAC](https://learn.microsoft.com/en-us/azure/role-based-access-control/overview) roles (e.g. Contributor, VM Administrator Login) are not permanently assigned — they must be activated on-demand through **[Privileged Identity Management (PIM)](https://learn.microsoft.com/en-us/entra/id-governance/privileged-identity-management/pim-configure)** in the Azure Portal. Some roles also require a second team member to approve before access is granted. This limits the blast radius of compromised credentials and enforces an audit trail for all privileged access.

## SSH Config Prerequisite (Ubuntu 22.04+ / new OpenSSH clients)

New [OpenSSH](https://www.openssh.com/) clients are incompatible with SSH certificates generated by `az ssh`. Before connecting, add the following to `/etc/ssh/ssh_config.d/azure-ssh.conf` (or to `~/.ssh/config`):

```
Host *
    PubkeyAcceptedKeyTypes +ssh-rsa-cert-v01@openssh.com
```

## Regular PIM Elevation

Perform these steps **before** attempting to SSH into the jumphost.

**Roles required for jumphost login:**

- `NAV Routing/Search Batch - prod - Reader`
- `NAV Routing/Search Batch - prod - Virtual Machine Administrator Login`
- `NAV Routing/Search Batch - prod - Azure Kubernetes Service Contributor Role`
- `NAV Routing/Search Batch - prod - Contributor`

For AKS cluster access from the jumphost, `Azure Kubernetes Service Cluster User Role` is also required.

**Step-by-step:**

1. Open the PIM activation page:
   [https://portal.azure.com/#blade/Microsoft_Azure_PIMCommon/ActivationMenuBlade/azurerbac](https://portal.azure.com/#blade/Microsoft_Azure_PIMCommon/ActivationMenuBlade/azurerbac)

2. Navigate to **My Roles → Azure resources**.

3. Find the role you need to activate (e.g. `Virtual Machine Administrator Login`) in the **Eligible assignments** list and click **Activate**.

4. In the activation dialog:
   - Enter a **reason** (e.g. "oncall incident investigation").
   - Set a **duration** (default is 4 hours; extend if needed for longer incidents).
   - Click **Activate**.

5. If the role requires approval, your request enters a pending state and an email is automatically sent to all team members with the same eligible assignment. Monitor your pending requests at:
   [https://portal.azure.com/#blade/Microsoft_Azure_PIMCommon/RequestMenuBlade/azurerbac](https://portal.azure.com/#blade/Microsoft_Azure_PIMCommon/RequestMenuBlade/azurerbac)

6. Once approved, you receive a confirmation email. The role appears under **My Roles → Azure Resources → Active assignments**.

7. After your work is done, deactivate the role from the Active assignments list to follow least-privilege hygiene.

## Approval Process

Any team member with the same **Eligible assignment** for a role can approve another team member's PIM request.

**To approve a pending request:**

1. Open the PIM approval page:
   [https://portal.azure.com/#blade/Microsoft_Azure_PIMCommon/ApproveRequestMenuBlade/azurerbac](https://portal.azure.com/#blade/Microsoft_Azure_PIMCommon/ApproveRequestMenuBlade/azurerbac)

2. Navigate to **Privileged Identity Management → Approve requests → Azure resources**.

3. Select the pending request and click **Approve**, providing a justification comment.

**Best practice:** Ask someone from **Team Stratus** (the owning team) to approve. Team Stratus members have the correct eligible assignments and context to approve safely. Ping the team on Slack rather than waiting for the automated email.

**Expected wait time:** During business hours, approval usually arrives within a few minutes after pinging a teammate on Slack. Outside business hours, follow the out-of-hours procedure below — NOC handles approval instead.

## Out-of-Hours PIM (NOC Contact)

Outside business hours, no team members may be available to approve your PIM request. In that case, **NOC approves the elevation**.

**Roles NOC can approve:**

| Role | Purpose |
|------|---------|
| `Virtual Machine Administrator Login` | SSH login to jumphost |
| `Azure Kubernetes Service Cluster User Role` | AKS access with user credentials (kubectl prompts for TomTom login) |
| `Azure Kubernetes Service Cluster Admin Role` | AKS access with admin credentials (`az aks get-credentials --admin`) |
| `Reader` | View-only access to all resources |
| `Contributor` | Full resource management, no role assignment |
| `Owner` | Full resource management including role assignment |

**Procedure:**

1. Activate the PIM elevation as normal (see Regular PIM Elevation above) — do this first so the request exists in the system.

2. Go to the **#noc-pim-request** Slack channel:
   [https://tomtomslack.slack.com/archives/CAZJP939P](https://tomtomslack.slack.com/archives/CAZJP939P)

3. Submit your request using the built-in **workflow form** pinned in the channel bookmarks. Do not post a free-form message — use the workflow.

4. NOC will approve the PIM request in the Azure Portal. You will receive a confirmation email once approved.

NOC workflow documentation: [Workflow for PIM Requests in the #noc-pim-request Channel](https://tomtom.atlassian.net/wiki/spaces/ITSM/pages/149633671)

## Accessing the Jumphost

### Method 1: Using the login script (recommended)

Requires the batch-service2-infra repo checked out with dependencies installed (see [prerequisites-tools.md](https://github.com/tomtom-internal/batch-service2-infra/blob/master/docs/prerequisites-tools.md)).

```bash
cd batch-service2-infra/jumphost

./jumphost-login.sh                          # show help
./jumphost-login.sh prod                     # log in to main jumphost VM
./jumphost-login.sh prod --backup-jumphost   # log in to backup jumphost VM
```

### Method 2: Direct az ssh (current az cli ssh extension >= 0.1.6)

```bash
az ssh vm --ip jumphost.prod.batch.tt4.nl -- -tA
```

### Method 3: az ssh config (compatible with older az cli ssh extension)

```bash
az ssh config --ip jumphost.prod.batch.tt4.nl -f /tmp/sshconfig-jumphost
ssh -F /tmp/sshconfig-jumphost -tA jumphost.prod.batch.tt4.nl
```

### After logging in

Configure your user environment (first time or after jumphost redeploy):

```bash
/opt/tomtom/jumphost-configure-user-environment.sh
```

Log in with your business Azure account:

```bash
az login --use-device-code
```

### Accessing AKS clusters from the jumphost

Use the provided bash aliases (available after configuring the user environment):

```bash
az-aks-get-credentials-prod-user aks21 westeurope
```

This sets the current kubectl context to `aks21-prod-westeurope`. After that, standard `kubectl` commands work against the cluster.

### Tools available on jumphost

[`az cli`](https://learn.microsoft.com/en-us/cli/azure/), [`terraform`](https://developer.hashicorp.com/terraform/docs), [`helm`](https://helm.sh/docs/), [`kubectl`](https://kubernetes.io/docs/reference/kubectl/), `kubectl-enter`, [`kubelogin`](https://azure.github.io/kubelogin/), [`azcopy`](https://learn.microsoft.com/en-us/azure/storage/common/storage-use-azcopy-v10)

## Break-Glass Scenario

Use break-glass **only when PIM is unavailable or broken** and normal `az ssh` login is impossible. It uses [HashiCorp Vault](https://developer.hashicorp.com/vault/docs)-signed SSH certificates and the jumphost's managed identity to grant Contributor access.

### Step 1: Sign your SSH key via Vault

1. Copy your private SSH key:

   ```bash
   cp ~/.ssh/your-work.key ~/.ssh/jumphost.key
   ```

2. Open the Vault break-glass signing page (allow pop-ups for `vault.tomtomgroup.com`):
   [https://vault.tomtomgroup.com/ui/vault/secrets/break-glass/sign/default?namespace=cit-compute-public](https://vault.tomtomgroup.com/ui/vault/secrets/break-glass/sign/default?namespace=cit-compute-public)

3. Sign in using the **OIDC Provider** button if prompted.

4. Click **More options** to reveal all fields.

5. Paste your **public SSH key** into the "Public Key" field.

6. Enter the subscription ID into the "Valid principals" field:
   ```
   2c48294f-c12f-4cd5-8c1a-b57f43e6fa43
   ```

7. Click **Sign**. Copy the resulting **Signed key** value.

8. Save the signed key to `~/.ssh/jumphost.key-cert.pub` (or as indicated by Vault's output).

### Step 2: SSH into the jumphost using the signed key

```bash
ssh -o PreferredAuthentications=publickey -tA -l azureuser jumphost.prod.batch.tt4.nl -i ~/.ssh/jumphost.key
```

### Step 3: Obtain Contributor access via managed identity (if PIM is completely unavailable)

Once on the jumphost, log in using its User Assigned Managed Identity to assume the Contributor role:

```bash
az login --identity \
  -u /subscriptions/2c48294f-c12f-4cd5-8c1a-b57f43e6fa43/resourceGroups/managed-identities-prod/providers/Microsoft.ManagedIdentity/userAssignedIdentities/jumphost
```

This grants Contributor access on the subscription through the jumphost VM's identity. Use this as a last resort only.

### Who to notify

- Inform the **on-call manager** / team lead that break-glass was used.
- Post in the team Slack channel with the reason and time of access.
- File an incident or post-mortem note documenting why normal PIM access was unavailable.

To find the managed identity resource ID, search for "Managed Identities" in the Azure Portal, locate the `jumphost` identity, and check its Properties for the full resource ID.

## Troubleshooting

### PIM activation stuck / pending with no approver available

- During business hours: ping a teammate on Slack and share the PIM approval link:
  [https://portal.azure.com/#blade/Microsoft_Azure_PIMCommon/ApproveRequestMenuBlade/azurerbac](https://portal.azure.com/#blade/Microsoft_Azure_PIMCommon/ApproveRequestMenuBlade/azurerbac)
- Out of business hours: use the **#noc-pim-request** Slack workflow (see Out-of-Hours PIM section above).
- If PIM itself is broken (Azure outage): use the break-glass scenario.

### SSH connection fails with no output

Stale host key entries can silently block SSH. Clear them and retry:

```bash
ssh-keygen -R jumphost.prod.batch.tt4.nl
ssh-keygen -R jumphost2.prod.batch.tt4.nl
```

Then retry the connection using Method 2 or Method 3 above.

### SSH key errors with new OpenSSH (Ubuntu 22.04+)

Add the following to `/etc/ssh/ssh_config.d/azure-ssh.conf` or `~/.ssh/config`:

```
Host *
    PubkeyAcceptedKeyTypes +ssh-rsa-cert-v01@openssh.com
```

### Jumphost VM itself is unresponsive

Redeploy the jumphost using the [Jenkins](https://www.jenkins.io/doc/) job:
[https://ci.dev.batch.tt4.nl/view/Batch/view/Prod/view/Jumphost/job/deploy_jumphost_prod/](https://ci.dev.batch.tt4.nl/view/Batch/view/Prod/view/Jumphost/job/deploy_jumphost_prod/)

After redeploy, the new VM is available at the same DNS address (`jumphost.prod.batch.tt4.nl`). If the primary jumphost is unreachable during redeploy, use the backup:

```bash
az ssh vm --ip jumphost2.prod.batch.tt4.nl -- -tA
```

### az login browser popup on jumphost

The jumphost environment may incorrectly detect a browser. Always pass `--use-device-code`:

```bash
az login --use-device-code
```
