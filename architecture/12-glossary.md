# 12. Glossary

| Term | Definition |
|------|------------|
| **Batch&Matrix / Batch&Matrix 1.2** | Three related TomTom APIs that accept large batches of queries and return aggregated results: Batch Search API, Batch Routing API, and Matrix Routing v1. Also called "classic Batch&Matrix". A single submission may contain up to 10,000 queries. Implemented in the `batch-service2-1.2` GitHub repository. |
| **Matrix Routing v2** | A separate, newer matrix routing product. It shares some infrastructure with Batch&Matrix 1.2 but is developed and operated independently. Implemented in the `batch-service2` GitHub repository. Many quota and processing rules that apply to Batch&Matrix 1.2 do not apply here. |
| **Batch item** | A single query inside a Batch Search or Batch Routing submission — for example, one route calculation request. A submission can contain up to 10,000 batch items. |
| **Matrix cell** | A single origin-destination pair inside a Matrix Routing submission. Analogous to a batch item; up to 10,000 cells per submission. |
| **front12** | The HTTP frontend component of Batch&Matrix 1.2. Receives client submissions and download requests, validates them, looks up quota information, and publishes messages to the Pulsar queue. It is the entry point into the Batch&Matrix processing pipeline. |
| **backend12** | The batch processor component of Batch&Matrix 1.2. Consumes messages from the Pulsar queue, calls the underlying services on behalf of the client, and writes the aggregated result to [Azure Blob Storage](https://learn.microsoft.com/en-us/azure/storage/blobs/storage-blobs-overview) when processing is complete. |
| **Underlying service** | Any TomTom API that Batch&Matrix calls to execute the queries inside a submission — for example, Routing API, Search/Geocoding API, or Reverse Geocoding API. Batch&Matrix fans individual items out to these services using the client's own API key, then aggregates the responses. |
| **Client key / API key** | The TomTom developer API key provided by the customer. Batch&Matrix uses this key when calling underlying services, so quota consumption counts against the customer's own quotas for those services. |
| **Customer token** | An internal identifier for the customer's APIM developer application (also called "developer app ID"). front12 logs it on every new submission. Use it to look up per-customer metrics in Batch&Matrix [Grafana](https://grafana.com/docs/grafana/latest/) and APIM Grafana dashboards. |
| **QPS (Queries Per Second)** | A rate quota that limits HTTP requests per second. Two independent QPS values matter in Batch&Matrix: (1) QPS to Batch&Matrix itself — controls how fast a client can submit and poll for results; (2) QPS to the underlying service — controls how fast Batch&Matrix can process items on behalf of a client. |
| **QPM (Queries Per Minute)** | A rate quota like QPS, but measured per minute. APIM enforces both QPS and QPM limits; exceeding either returns HTTP 429 responses. |
| **[PIM (Privileged Identity Management)](https://learn.microsoft.com/en-us/entra/id-governance/privileged-identity-management/pim-configure)** | An Azure Active Directory feature that grants time-limited elevated access to production systems. Request PIM activation before accessing production Kubernetes clusters or other restricted resources. Access requires approval and expires automatically. See the Confluence runbooks for how to request and approve PIM. |
| **Jumphost** | A hardened virtual machine that acts as a secure gateway to production infrastructure (AKS clusters, databases, Pulsar toolset). Connect to the jumphost first via VPN, then from the jumphost to internal services. Accessing the jumphost itself also requires PIM. |
| **[Pulsar (Apache Pulsar)](https://pulsar.apache.org/)** | The distributed message queue that decouples front12 from backend12 in Batch&Matrix 1.2. When a client submits a batch, front12 publishes a message to a Pulsar topic; backend12 consumers pick it up and begin processing. Each customer has dedicated topics (e.g., SEARCH2, ROUTING1, MATRIX1). The topic backlog size shows how many submissions are waiting to be processed. |
| **NOC** | Network Operations Center — the TomTom team responsible for first-line monitoring and incident triage. NOC engineers follow runbooks and escalate to on-call developers when they cannot resolve an issue. |
| **APIM / [Apigee](https://cloud.google.com/apigee/docs)** | API Management — the quota and authentication enforcement layer in front of all TomTom public APIs. Apigee is the underlying technology; APIM is the broader platform. Enforces QPS and QPM limits, validates API keys, and routes traffic. Quota configuration is governed by commercial contracts and cannot be changed by the Batch&Matrix team directly. |
| **[Traffic Manager (Azure Traffic Manager)](https://learn.microsoft.com/en-us/azure/traffic-manager/traffic-manager-overview)** | A DNS-based load balancer that distributes client traffic across geographic regions (e.g., EU and US). If a region becomes unhealthy, disable it via a runbook to direct all traffic to the remaining region. |
| **Multi-quota mapping** | A Batch&Matrix 1.2 configuration that maps each underlying-service endpoint to its Apigee "ratelimiter Quota Name". Endpoints that share a quota name in Apigee (e.g., geocode and structured geocode) are grouped so Batch&Matrix respects the shared limit correctly. Configured in `backend/src/main/resources/application.yml` under `rate-limiter.multi-quota-mapping`. |
| **Backlog** | The number of Pulsar messages (batch submissions) queued in a topic and not yet picked up by backend12. A growing backlog with no corresponding throttle events indicates consumers may be stuck, and triggers a Grafana alert. |
| **Break-glass scenario** | An emergency procedure for when normal access paths (PIM, jumphost) are unavailable or too slow. Grants immediate, unrestricted access to production systems for incident resolution. All use is logged and audited. See the Confluence runbook for details. |
| **Fire drill** | A planned exercise where the on-call team practices an emergency runbook (e.g., disabling a Traffic Manager region or rotating an API key) in a non-production environment. Verifies that the procedure works and that engineers are comfortable with it before a real incident. |
| **Tracking ID** | A unique identifier assigned to each Batch&Matrix submission. front12 logs it alongside the batch ID, API key, and customer token. Use it to correlate a specific submission across logs and metrics during incident investigation. |
| **Synchronous mode** | A submission mode where the client waits for the HTTP response to contain the completed result. Suitable for small, low-latency submissions. The connection stays open until processing finishes or a timeout occurs. |
| **Asynchronous mode** | A submission mode where the client receives an immediate HTTP 202 acknowledgement, then polls a separate download endpoint to retrieve the result once processing is complete. Recommended for large submissions or when results are not needed right away. |
| **[AKS (Azure Kubernetes Service)](https://learn.microsoft.com/en-us/azure/aks/what-is-aks)** | The managed [Kubernetes](https://kubernetes.io/docs/home/) platform where Batch&Matrix components (front12, backend12) are deployed. Accessing production AKS clusters requires PIM activation and jumphost access. |
